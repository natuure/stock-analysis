# 데이터 파이프라인 — Python 스크립트·수집 로직·데이터 구조

## 전체 데이터 흐름 (요약)

```
python 뉴스분석.py  ← 장마감 후 실행
    ├── FinanceDataReader로 전종목 + 코스피/코스닥 지수 수집
    ├── MongoDB stock_data 컬렉션 저장
    ├── 토스증권 캔들 미리 조회 → MongoDB candles 컬렉션 캐싱 (KIS 장애 시 폴백용)
    ├── Naver 뉴스 API 검색
    └── 분석결과/뉴스데이터_YYYYMMDD.json 저장
         ↓
Claude Code "분석해줘" 요청 → 분석결과/분석결과_YYYY-MM-DD.json 생성
         ↓
python 저장분석.py → MongoDB ai_analysis 컬렉션 저장

python 주간분석.py  ← 아무 때나 실행 가능 (별도 흐름, 위 일간 분석과 무관)
    ├── FDR로 코스피/코스닥 주간(월~금) 변동률 계산
    ├── 뉴스분석.py를 import해 KIS 통합(KRX+NXT) 보강 재사용 → 주간 거래대금·등락률
    │     상위 50도 같이 계산(2026-06-27 추가) → MongoDB weekly_indices 컬렉션 저장
    └── FDR로 ETF 유니버스(fdr.StockListing('ETF/KR')) + 종목별 주간 등락률 계산
          → 최소 AUM(100억) 필터 후 상위 15 → MongoDB weekly_indices.etfRank로 $set
          (2026-07-04 도입, 원래 별도 ETF분석.py였던 것을 2026-07-06부터 여기로 흡수)
         ↓
웹앱: /api/getData, /api/getAnalysis 는 MongoDB만 읽어서 화면 표시
     (달력 주차(W##) 칸의 주간 변동률은 /api/getData가 weekly_indices를 같이 읽어 내려주고,
     그 주차를 클릭하면 /api/getData?week=로 그 주의 거래대금·등락률 상위 50 + ETF 등락률
     상위 15(etfRank)도 함께 지연 조회함 — 카테고리 비중 도넛과 주간 종목 데이터 표 사이에 표시)
     /api/candles 는 KIS Open API를 직접 호출(실시간) → 실패 시에만 candles 캐시(토스) 폴백

python 종목분석.py 종목명  ← 단일 종목 재무제표·현재가 수집, 수동 실행
    ├── DART Open API로 최근 3~4개년 사업보고서 + 올해 진행 분기/반기보고서 수집
    ├── KIS Open API(UN, KRX+NXT 통합)로 현재가·시가총액·발행주식수 수집
    ├── 종목분석결과/{종목명}_{YYYYMMDD}.json 저장
    ├── MongoDB company_analysis 컬렉션 저장(_id=종목코드, upsert) → 웹앱 "종목 분석" 탭에서
    │     종목명 검색(자동완성)하면 /api/getCompanyOverview가 바로 조회해 표시 (2026-06-25)
    └── Claude Code가 이 JSON + dart-mcp/웹 검색으로 서술형 리포트(.docx) 작성 (수동, /종목분석
          스킬 참고, 웹앱 표시와는 별개의 산출물)

python 주도주분석.py [YYYY-MM-DD]  ← 뉴스분석.py 다음에 실행(수동, Claude 미사용), 날짜
생략 시 오늘
    ├── MongoDB stock_data의 그날 vol+rate(거래대금·등락률 상위 50, 최대 100개 중복 제거)를
    │     읽어 종목분석.py의 analyze_one()을 일괄 호출(2026-06-27)
    └── 종목별 company_analysis에 저장된 latest_report가 DART 실제 최신 보고서와 같으면
          스킵(이미 최신), 다르면(새 분기/반기/사업보고서 발생) 그 종목만 재분석
```

> ⚠️ **토스증권 API는 IP 허용 목록 기반**이라 Vercel 서버리스 함수(유동 IP)에서 직접 호출하면
> 막힌다. 반면 **KIS(한국투자증권) Open API는 IP 제한이 없음을 확인**(2026-06-20)했고, 캔들 조회는
> `api/candles.js`가 KIS를 직접 호출한다. 토스 호출은 고정 IP인 로컬 `뉴스분석.py`에서만 계속하며,
> 그 결과는 KIS 호출이 실패했을 때만 쓰이는 폴백 캐시다.

## Python 스크립트

### 뉴스분석.py
```bash
python 뉴스분석.py   # 장마감 후 실행. 오늘 날짜 기준 KRX 전종목 자동 수집
```
- FinanceDataReader(`fdr.StockListing('KRX')`)로 전종목 시세 1회 호출 (KONEX 제외)
- 거래대금 상위 50 / 등락률 상위 50(거래대금 300억↑ 대상) 산출 — **KIS 통합(KRX+NXT)
  데이터로 보강** (2026-06-21 도입, 아래 "KIS 통합 거래대금·등락률 보강" 참고)
- 코스피·코스닥 지수 종가/등락률도 같이 수집 (`fetch_indices()`, `fdr.DataReader('KS11'|'KQ11')`)
- MongoDB `stock_data` 컬렉션에 저장 (웹앱 달력 초록 점 자동 표시)
- 거래대금+등락률 종목(최대 100개)의 토스증권 일봉 캔들 85개를 미리 조회해 MongoDB `candles`에 캐싱
  (토스 API는 IP 허용 목록 기반이라 고정 IP인 로컬에서만 호출, Vercel은 직접 호출하지 않음.
  2026-06-20부터는 `api/candles.js`가 KIS를 우선 호출하므로, 이 캐싱은 KIS 장애 시에만 쓰이는
  폴백 데이터가 됨 — 자세한 내용은 아래 "KIS 일별 캔들 조회(api/candles.js)" 참고)
- `분석결과/뉴스데이터_YYYYMMDD.json` 생성
- Naver 뉴스 쿼리: `{name} 특징주`, `{name} 급등 이유`, `{name} 상승 배경` 등 8개
- 날짜 기준 검색: 파일날짜 ~ 오늘 (3일 이상 지난 경우 파일날짜+3일로 제한)
- 당일 기사 우선 정렬

### 주간분석.py (2026-06-21 도입, 2026-06-21 범위 축소, 2026-06-27 거래대금·등락률 상위 50 추가)
```bash
python 주간분석.py   # 아무 때나 실행 가능 (장마감 후가 자연스러움)
```
- `fdr.DataReader('KS11'|'KQ11')`로 최근 21일치(이번 주 + 비교 기준 지난 주 확보용 여유)만
  받아서 **가장 최근 1주(월~금)치만** 계산 — 과거 주차는 다시 계산하지 않음(이미 저장된
  과거 주는 그대로 유지됨)
- `monday_of(today)`로 이번 주 월요일을 구하고, 그 주 마지막 거래일 종가 vs 지난 주
  마지막 거래일(보통 금) 종가를 비교해 `{close, change, changeRate}` 계산. 이번 주에
  아직 거래일이 없으면(주말·휴일에 실행 등) 한 주 전으로 자동 이동
- 주차 키는 `f'{d.year}-W{iso_week}'` 형태(예: `2026-W25`) — `src/utils.js`의
  (현재는 `Calendar.jsx` 내부에 로컬로 있는) ISO 주차 계산과 동일한 규칙이어야
  웹앱이 키를 매칭할 수 있음. **숫자가 아닌 문자열이라 사전식 정렬로는 시간순이
  안 됨** (`"2026-W9"` > `"2026-W25"`)
- MongoDB `weekly_indices` 컬렉션에 해당 주차 1건만 upsert
- 웹앱 `/api/getData`가 이 컬렉션 전체를 같이 읽어 `weeklyIndices`로 내려주고,
  달력의 **주차(W##) 칸**(2026-06-22부터 — 그 전에는 토요일 칸이었음)이 그 주(월~금)의
  코스피/코스닥 변동률을 표시함(`Calendar.jsx`가 그 주 월요일 날짜로 주차를 계산해 매칭)
- 처음 도입 시점에 과거 ~76주를 1회 백필해둔 상태라 과거 달도 정상 표시됨. 더 과거
  주차가 비어있으면 `LOOKBACK_DAYS`를 늘려 그 주만 따로 1회 실행해 백필하면 됨.
- **거래대금·등락률 상위 50도 같이 산출**(2026-06-27 추가, 사용자 요청 — "뉴스분석.py가
  하루치를 하는 것처럼 일주일치도 가능한가") — `뉴스분석.py`를 `import`해 KIS 통합
  (KRX+NXT) 보강 로직(`get_kis_token`/`_fetch_kis_daily`/`fetch_market_data`/
  `RATE_PRECHECK_MIN_AMOUNT`/`RATE_MIN_AMOUNT`/`UPPER_LIMIT_RATE`)을 그대로 재사용한다
  (`주도주분석.py`가 `종목분석.py`를 임포트하는 기존 패턴과 동일, `if __name__=='__main__'`
  가드라 임포트 시 일간 `main()`은 실행 안 됨).
  1. **`resolve_target_week()`**: KOSPI 지수로 이번 주(또는 거래일 없으면 가장 최근 완결
     주)의 월요일과 **실제 거래일 목록**(공휴일 제외)을 계산 — `weekly_change()`와는 별개
     함수로 분리해 지수 계산 로직은 건드리지 않음.
  2. **`fetch_weekly_market_data(trading_dates)`**: `뉴스분석.fetch_market_data()`로
     KONEX·스팩 제외한 전종목 유니버스를 얻고, 종목별로
     `fdr.DataReader(code, 그 주 시작 10일 전, 마지막 거래일)`을 1회 호출(전체 ~2,875개
     종목 기준 약 3분, 종목당 0.058초로 직접 측정함)해 주간 거래대금 근사치를 계산한다.
     **`fdr.DataReader`의 종목별 히스토리에는 `StockListing` 스냅샷에 있는 `Amount`
     (거래대금) 필드가 없어**(직접 확인) `Volume×Close` 일별 합산으로 근사하고, 등락률
     근사치는 주 시작 전 마지막 종가 대비로 계산한다. 이 근사치는 KIS 보강 후보 풀을
     추리는 데만 쓰고 최종 순위는 KIS 통합 데이터로 매긴다.
  3. **임계값은 그 주 실제 거래일 수에 비례 스케일링**(고정값 아님) —
     `뉴스분석.RATE_PRECHECK_MIN_AMOUNT`(30억)·`뉴스분석.RATE_MIN_AMOUNT`(300억)에 그 주
     실제 거래일 수를 곱함(5거래일 주 → 150억/1500억, 휴일로 4거래일인 주 → 120억/1200억).
     선형 근사라는 한계는 있지만, 일간 임계값 자체도 경험적으로 튜닝된 값이라 가장 단순한
     모델로 채택.
  4. **`enrich_weekly_with_kis(...)`**: 후보 종목별로 `뉴스분석._fetch_kis_daily(token,
     code, 'UN', 그 주 마지막 거래일, lookback_days=15)`를 1회 호출(날짜 범위를 한 번에
     반환하는 기존 함수를 그대로 재사용 — 일간은 "오늘 1행"만 읽었지만 여기선 그 주에
     속하는 여러 행을 합산)해 거래대금·거래량 합계와 주 마지막 거래일 통합(UN) 종가를
     구하고, `뉴스분석._fetch_kis_daily(..., 'J', ...)`로 주 시작 전 마지막 거래일의 KRX
     공식 종가를 등락률 기준선으로 따로 받는다(일간과 동일한 이유 — 등락률은 항상 KRX
     공식 전일종가 기준, UN 종가 기준이 아님). 종목별 호출 실패는 그 종목만 FDR 근사치로
     폴백(명단에서 빠지지 않음), KIS를 전혀 못 쓰면(키 없음·토큰 실패) `None`을 반환해
     `main()`이 그 실행에서는 vol/rate를 저장하지 않고 건너뛰게 한다.
  5. **등락률 표의 "상한가" 표시는 "그 주에 하루라도 일간 상한가(29.5%↑)를 친 종목"으로
     재정의**(2026-06-27, 사용자 확정) — 주간 등락률 합산값 자체에는 일간의 ±30% 한도
     개념이 그대로 적용되지 않으므로, 이미 받아온 그 주 UN 일별 종가들을 연속으로 비교한
     일간 등락률(주 첫째 날만 위 4번의 J 기준선과 비교)을 계산해 그중 하루라도 29.5%
     이상이면 `isUpperLimit: true` — 추가 KIS 호출 없이 기존에 받은 데이터로 판정한다.
  6. **`get_previous_week_vol_ranks(week_key)`**: 직전 주의 거래대금 순위를 가져와
     `prevRank`를 채움(일간의 `get_previous_vol_ranks()`와 동일한 목적). `weekly_indices`
     의 `_id`는 위에서 언급한 대로 문자열 정렬이 시간순이 아니므로, 전체 문서를 가져와
     `(year, iso주차)` 튜플로 직접 비교해서 직전 주를 찾는다. 첫 실행이거나 직전 주에
     `vol`이 없으면 빈 딕셔너리 반환(일간과 동일).
  7. 결과는 `weekly_indices` 문서에 `vol`(50개)·`rate`(50개)·`lastTradingDate`(그 주
     마지막 실제 거래일, "YYYY-MM-DD") 필드로 같이 저장(`$set`이라 기존 `kospi`/`kosdaq`은
     그대로 둠). `high60Rate`(60일 신고가 대비)·토스 캔들 캐싱·네이버 뉴스 수집은 이번
     범위에서 제외 — `Tables.jsx`는 `showHigh60Rate={false}`를 받아 그 열 자체를 표에서
     숨긴다(2026-06-27, `<Tables>` prop 추가 — 처음엔 필드가 없을 때 `-`로만 표시했다가
     빈 열을 아예 숨기는 쪽으로 변경, [FRONTEND.md](FRONTEND.md) 참고).
  8. **`fetch_weekly_category_map(trading_dates)` + `attach_categories(...)`**(2026-06-27
     추가, 사용자 요청 — "주간분석에서도 카테고리 비중 도넛 차트") — 주간분석.py는
     Claude/Codex를 호출하지 않아 직접 카테고리를 매기지 못하므로, 그 주 거래일들의
     일간 `ai_analysis`(이미 분석·저장된 것)에서 종목명별 카테고리를 모아 재사용한다.
     `ai_analysis`를 `_id`(날짜 문자열) 기준으로 그 주 날짜만 조회 → `analysis.거래대금`/
     `등락률`을 날짜 오름차순으로 순회하며 `{종목명: {카테고리, 신규카테고리후보}}` 맵을
     덮어쓰기로 구성(그 주 안에서도 더 최근 날짜의 분류가 우선 — 일간의 "분석 시점 기준"
     원칙을 주 단위로 적용)한 뒤, `vol`/`rate` 각 항목에 종목명으로 매칭해 `카테고리`(+
     해당 시 `신규카테고리후보`)를 채운다. **매칭 안 되는 종목은 필드 자체를 안 붙임**
     (그 주 어느 날의 일간 상위 50에도 없었던 종목 등) — 프론트의 `aggregateByCategory()`가

```bash
python 주도주분석.py 2026-06-25  # 특정 날짜로 실행
```

- `뉴스분석.py`가 그날 MongoDB `stock_data`에 저장한 `vol`(거래대금 상위 50) + `rate`

- **이미 최신이면 건너뛴다**: `company_analysis`에 저장된 `latest_report`(연도·보고서
  종류)가 DART에 실제로 존재하는 최신 보고서와 정확히 같으면 재분석하지 않고 스킵 —
  같다는 건 `build_fetch_plan()`이 만드는 조회 범위도 똑같다는 뜻이라 다시 돌려도 같은
  결과만 나옴. 다르면(그 사이 새 분기·반기·사업보고서가 올라왔으면) 그 종목만 재분석해
  최신화한다. 한 번도 분석한 적 없는 종목은 그냥 새로 분석.
- **종목명 → corp_code 매칭은 이름 기반 퍼지매칭(`find_corp_code()`)이 아니라, `vol`/
  `rate`에 이미 있는 정확한 KRX 코드로 `_dart_corp_codes.json`의 `stock_code`를 역매핑**
  한다(직접 확인: "현대차"라는 표시명은 DART 정식명 "현대자동차"와 전혀 안 겹쳐 퍼지매칭도
  실패하지만, 코드 `005380`으로는 정확히 찾아짐 — 사람이 타이핑한 검색어의 오타를
  흡수하려고 만든 이름 매칭은 이미 정확한 코드를 아는 이 배치 스크립트에는 안 맞는 도구).
- **우선주(예: "삼성전자우", "현대차2우B")는 코드로도 이름으로도 DART corp_code 매핑에
  없음**(DART corpCode.xml은 발행회사 단위라 우선주 티커를 따로 안 줌, 직접 확인) — 코드
  매칭이 실패하면 우선주 접미사(`...우`, `...우B`, `...2우B` 등)를 뗀 이름의 **정확
  일치만** 한 번 더 시도한다(`삼성전자우`→`삼성전자`, `LG화학우`→`LG화학`처럼 흔히
  공통주 정식명과 약식 우선주명이 일치하는 경우는 해결됨). **이때 부분일치 폴백은 의도적으로
  안 씀** — 직접 테스트해보니 "현대차2우B"를 "현대차"로 뗀 뒤 부분일치를 타면 무관한
  "현대차증권"에 잘못 매칭됨("현대차"가 "현대차증권"의 부분문자열이라서, `sk하이닉스`→
  `이닉스` 사고와 같은 패턴) — 정확 일치만 허용해 틀린 회사로 잘못 매칭될 위험을 없애는
  대신, "현대차2우B"처럼 약식명이 정식명과 전혀 안 겹치는 경우는 못 찾고 건너뛴다(알려진
  한계, DART 계정명 변칙들을 다 못 잡는 것과 같은 성격).
- **KIS 접근토큰은 배치 전체에서 1번만 발급해 재사용**한다 — `종목분석.py`는 원래 "1분당
  1회 제한에 거의 안 걸리는 단독 실행" 전제로 토큰을 캐싱하지 않는데, 100개 종목을 그대로
  돌리면 거의 다 토큰 발급 실패로 `quote` 없이 끝남. `get_kis_token()`을 한 번만 호출해
  `analyze_one()`에 넘겨준다(그래도 실패하면 그 배치는 `quote` 없이 진행 — 화면에 보이는
  PER/PBR은 어차피 검색 시점에 KIS로 실시간 재조회되는 값이라 영향 없음).
- 실행 로그 마지막에 "분석 N건 / 스킵 N건 / 실패·건너뜀 N건" 요약이 출력됨.

## 데이터 수집 (FinanceDataReader)

`fdr.StockListing('KRX')` 1회 호출로 KRX 전종목(KOSPI/KOSDAQ, KONEX 제외)의
`Code, Name, Market, Close, Changes, ChagesRatio, Volume, Amount, Marcap`을 가져옴.
- **거래대금 상위 50**: `Amount` 내림차순
- **등락률 상위 50**: `Amount >= 300억` 필터 후 `ChagesRatio` 내림차순
- **상한가 판정**: `changeRate >= 29.5` (`isUpperLimit: true`)
- **전일 순위(prevRank)**: MongoDB에서 직전 거래일 `stock_data` 문서를 조회해 종목코드로 매칭 (없으면 NEW)
- **체결강도**: FDR로 계산 불가하여 컬럼 자체 제거됨 (HTS 엑셀 시절에는 있었음)
- **ETF/ETN/스팩 제외** (2026-06-21): `fetch_market_data()`에서 이름에 `스팩|기업인수목적`이
  들어간 행을 제외함. ETF/ETN은 직접 확인해본 결과 `fdr.StockListing('KRX')` 자체가 보통주/
  우선주만 반환해서(KODEX 200 등 주요 ETF 코드도 안 잡힘) 별도 필터가 필요 없었음 — 스팩만
  실제로 섞여 있었음(`Dept` 컬럼의 `SPAC(소속부없음)` 태그만으론 전부 안 잡혀서 이름 패턴
  사용, 관리종목으로 전환된 스팩까지 잡아냄). 기존 MongoDB `stock_data`/`ai_analysis`에 이미
  들어가 있던 스팩 1건(2026-06-19, 메리츠제2호스팩)도 같이 제거 + rank 재정렬함.

> ✅ **해결됨 — NXT(대체거래소) 거래량 포함 여부** (2026-06-19 미확인 → 2026-06-21 해결):
> FDR `StockListing('KRX')`는 NXT 체결분을 포함하지 않는 KRX 단독 수치임을 KIS Open API로
> 직접 확인했다(`FID_COND_MRKT_DIV_CODE`를 `J`/`NX`/`UN`으로 바꿔 같은 종목·날짜를 비교,
> `UN`=`J`+`NX` 정확히 일치). FDR 자체를 고칠 수는 없지만, 거래대금·등락률 상위 50을
> KIS 통합(UN) 데이터로 재보강하는 방식으로 해결함 — 아래 "KIS 통합(KRX+NXT) 거래대금·
> 등락률 보강" 참고.

### 60일 신고가 대비 등락률 (`high60Rate`, 2026-06-21 도입)
- 거래대금·등락률 표의 "거래량" 열을 대체. `cache_candles()`가 이미 받아온 캔들(종목당 85개)에서
  첫 60개(`HIGH60_WINDOW`)의 **종가**(`closePrice`) 중 최고값을 "60일 신고가"로 보고
  `(현재가-60일최고가)/60일최고가*100`을 계산 — 캔들 캐싱과 같은 호출로 처리되어 API 추가
  호출이 없다. intraday 고가(`highPrice`)가 아니라 종가 기준(2026-06-21 결정 — 일시적
  꼬리(wick)를 신고가로 치지 않기 위함).
- 0%를 상한으로 클램프(현재가가 60일 신고가 이상이면 0%) — **토스 캔들의 종가는 KRX 단독이
  아니라 KIS UN(NXT 포함 통합가)과 일치함을 직접 확인**(2026-06-21, 000660의 2026-06-18
  종가가 토스·KIS UN 모두 2,699,000으로 일치, 같은 날 FDR만 KRX 단독인 2,685,000 — FDR이
  아니라 토스도 KRX 단독이라고 잘못 적어뒀던 걸 정정함). 그래도 토스와 KIS 간 반영 시점이
  미세하게 어긋날 가능성에 대비해 클램프는 유지.
- 토스 캔들 fetch 실패 종목은 `high60Rate: null`(프론트에서 '-' 표시).
- `cache_candles(vol, rate, date)` 호출을 `save_to_mongodb(...)`보다 먼저 하도록 `main()` 순서를
  바꿔서, 계산된 `high60Rate`가 vol/rate 딕셔너리에 채워진 뒤 저장되게 함.
- 과거 06-15~19도 1회성 스크립트로 백필함(2026-06-21) — 그 날짜들에 이미 캐싱돼 있던 `candles`
  컬렉션(해당 날짜 기준 85개)을 그대로 재사용해 API 호출 없이 계산. 종목별로 그 날짜의 캔들이
  캐싱 안 돼 있던 소수(06-15 1개, 06-17 2개)는 `high60Rate: null`로 남음.
- 계산 기준을 intraday 고가에서 종가로 바꾼 뒤(2026-06-21) 06-15~19를 같은 방식으로 재백필함
  — 필드 구조는 안 바뀌었지만 값 자체가 달라진 콘텐츠 변경이라, 스팩 제거·등락률 기준선
  수정 때와 동일한 이유로 `CACHE_VERSION`을 한 번 더 올렸다(7→8).

### KIS 캔들 조회 — 일봉/주봉 (`api/candles.js`, 2026-06-20 도입, 주봉은 같은 날 추가)
- 종목 클릭 시 Vercel이 **KIS(한국투자증권) Open API를 실시간으로 직접 호출**한다.
  토스와 달리 KIS는 IP 허용 목록 제한이 없음을 확인했음 (Vercel 프리뷰 배포에서 토큰 발급·조회
  성공으로 검증, 2026-06-20).
- 엔드포인트: `GET /uapi/domestic-stock/v1/quotations/inquire-daily-itemchartprice`

  - `W`(주봉): 캔들 75개, `FID_INPUT_DATE_1`은 대상일 기준 540캘린더일 전(75주 확보용 여유)

- KIS 호출이 실패하면(네트워크 오류, 상장폐지 등) **일봉(`D`)일 때만** 기존 토스 캐시(`candles`
  컬렉션)로 폴백한다 — 이 캐시는 위 "토스증권 캔들 캐싱" 절차로 여전히 매일 채워지지만 일봉만
  보관하므로, 주봉(`W`)은 KIS 실패 시 폴백 없이 빈 배열을 반환한다.
- 이 전환으로 "그날 거래대금/등락률 상위 50에 든 종목만 캔들 조회 가능"하던 한계가 사실상
  해소됨 (KIS는 캐싱 없이 아무 종목이나 즉시 조회 가능).

### KIS 통합(KRX+NXT) 거래대금·등락률 보강 (`뉴스분석.py`, 2026-06-21 도입)
- FDR `StockListing('KRX')`는 KRX 거래소 단독 수치라 대체거래소 NXT(넥스트레이드)에서 체결된
  거래량·거래대금·종가가 빠져 있음을 KIS로 직접 검증함: `inquire-daily-itemchartprice`의
  `FID_COND_MRKT_DIV_CODE`를 `J`(KRX만)/`NX`(NXT만)/`UN`(통합)으로 바꿔 같은 종목·날짜를
  조회하면 **`UN` = `J` + `NX`가 거래량·거래대금 모두 정확히 일치**(삼성전자로 검증).
  종가도 더 늦게 마감하는 NXT 쪽 값이 그날의 진짜 최종 종가로 나옴.
- 종목 1개당 1~2회 호출해야 해서 전종목(약 2,696개)을 다 보강하긴 비효율적(추정 26~35분) —
  대신 **2단계 필터**로 후보를 추린다(2026-06-24 재설계):
  1. **1단계(사전 후보 풀)**: `RATE_PRECHECK_MIN_AMOUNT`(거래대금 30억 이상, FDR 단독 기준,
     평소 500개 안팎)로 넓게 추려 KIS로 보강한다(`enrich_with_kis()`).
  2. **2단계(최종 컷오프)**: 보강(또는 NXT 미거래 시 FDR 폴백) 후의 **통합 거래대금**이
     `RATE_MIN_AMOUNT`(300억) 이상인 종목만 남겨서 그중 등락률 상위 50을 뽑는다
     (`build_vol_rate_from_enriched()`). 거래대금 상위 50은 이 컷오프 없이 보강된 후보
     전체에서 그대로 뽑는다(원래도 최소 거래대금 기준이 없는 "무조건 상위 50"이라 영향 없음 —
     오늘 데이터 기준 거래대금 50등 컷오프 자체가 1,113억원으로 300억보다 훨씬 높음).
  - 1단계를 `RATE_MIN_AMOUNT`(300억)가 아니라 훨씬 낮은 `RATE_PRECHECK_MIN_AMOUNT`(30억)로
    잡는 이유: FDR `Amount`는 KRX 단독 거래대금이라, NXT 비중이 큰 종목은 실제(KRX+NXT
    통합) 거래대금이 300억을 넘어도 FDR 기준으론 한참 못 미칠 수 있다 — 2026-06-24 직접
    확인(에이프릴바이오 FDR 206.6억→KIS통합 794.6억/+285%, 로켓헬스케어 FDR 40.1억→KIS통합
    370.3억/+823%, 삼천당제약 FDR 297.7억→KIS통합 520.0억/+75%). 1단계 기준에서 300억을
    그대로 쓰면 이런 종목은 KIS 보강 자체를 못 받고 등락률 상위 50에서 통째로 누락된다.
- KIS 토큰은 `api/candles.js`의 `getKisToken`과 동일한 MongoDB `kis_token` 컬렉션을
  공유해서 캐싱한다(발급 1분당 1회 제한을 Vercel 쪽과 같이 지켜야 함).
- **UN 모드의 함정**: NXT에서 전혀 거래되지 않는 종목(우선주 등 일부)은 `UN`이 빈
  `output2`를 반환함(직접 확인). 이 경우 **`J`로 재조회하지 않고 바로 FDR(KRX 단독) 값으로
  폴백**한다 — `J`로 다시 KIS를 호출해도 FDR과 같은 KRX 단독 값일 뿐이라(직접 검증함),
  UN에 데이터가 있는 종목만 KIS를 쓰고 없는 종목은 KIS를 추가로 부르지 않는다(불필요한
  API 호출·호출 제한 소모 방지, 2026-06-21 결정).
- **등락률 기준선은 전일 UN 종가가 아니라 전일 KRX 단독(J) 종가**(2026-06-21 수정): 실제
  HTS 화면(SK하이닉스/삼성전자/삼성전기/SK스퀘어)과 대조해보니 현재가는 통합(UN) 최종가가
  맞는데 등락률/대비는 달랐다 — 예) SK하이닉스 06-19: HTS 대비 +70,000(전일 **J** 종가
  2,685,000 기준)인데, 전일 **UN** 종가(2,699,000) 기준으로 계산하면 +56,000이 나와서
  틀림. KRX의 공식 종가가 익일 가격제한폭 등 규제 기준이 되는 "전일종가"라서, NXT에서
  더 늦게 체결된 가격과 무관하게 등락률 기준선은 항상 KRX 단독 종가를 쓰는 것으로 보임.
  `fetch_kis_prev_close()`가 별도로 `J` 모드를 조회해 전일 종가를 가져옴 — 성공 종목당
  KIS 호출이 1회(UN)에서 2회(UN+J)로 늘어남.
- **호출 제한(`EGW00201`, "초당 거래건수를 초과하였습니다")**: 종목 사이 0.12초씩 쉬어도
  실제로 종종 걸림(직접 확인) — 걸리면 1초 쉬고 최대 3회 재시도한다.
- 종목 단위 호출 실패(상장정지·데이터 지연 등)는 그 종목만 FDR(KRX 단독) 값으로 폴백해서
  명단에서 빠지지 않게 한다. KIS 전체를 못 쓰면(키 없음·토큰 발급 실패) 기존 FDR 전용
  함수(`build_vol_list`/`build_rate_list`)로 통째로 폴백한다.
- 검증 결과(2026-06-19 기준, 180개 후보): UN에 데이터가 있는 133개는 KIS 성공·NXT
  미거래로 UN이 빈 47개는 FDR 폴백. SK하이닉스 거래대금이 FDR 단독 20.8조원 → KIS
  통합 34.7조원으로, 거래대금·등락률 상위 50 종목 구성 자체가 FDR 단독과 달라짐
  (NXT 비중이 큰 종목이 순위 상승).
- **한계**: 1단계 후보군 자체가 FDR(KRX 단독) 거래대금 30억(`RATE_PRECHECK_MIN_AMOUNT`)
  기준으로 뽑히므로, 이보다 더 작은 FDR 거래대금에서 극단적인 NXT 비중(2026-06-24 기준
  관찰된 최대치 +823%보다 훨씬 큰 경우)을 가진 종목이 나오면 이론상 여전히 누락될 수 있다
  (전종목을 다 보강하지 않는 한 100% 보장은 불가능한 트레이드오프). 30억은 2026-06-24
  실제 누락 사례(가장 작은 게 37.4억) 기준으로 여유를 둔 값.
- 과거 5개 날짜(06-15~19)도 1회성 스크립트로 백필함(2026-06-21). FDR `StockListing`은
  "오늘" 스냅샷만 줘서 과거 날짜의 RATE_MIN_AMOUNT 기준 후보군을 새로 못 뽑으므로, **이미
  저장돼 있던 그 날짜의 거래대금·등락률 top50 합집합(날짜당 53~80개)을 후보로 재사용**해
  KIS로 재계산·재정렬했다. `prevRank`는 이번 백필 배치 내에서 재정렬된 직전 날짜 순위를
  다시 따라가도록 재계산(06-15만 예외 — 그 이전 날짜가 백필 범위 밖이라 기존 값 유지 불가,
  prevRank 없음). 상장주식수(시가총액 계산용)는 단기간에 거의 안 바뀌므로 오늘자 FDR
  값을 그대로 재사용. 06-15~17은 원래 `vol`이 50개가 아니라 30개만 저장돼 있던 기존
  이력 버그도 이번 백필로 50개로 메워짐. `CACHE_VERSION`은 4→5로 한 번 더 올렸다(데이터
  내용만 바뀐 경우도 캐시 무효화가 필요해서, 스팩 제거 때와 동일한 이유).
  → 위 "등락률 기준선" 수정 직후 같은 방식으로 한 번 더 재백필함(`CACHE_VERSION` 5→6).

```
analysis_dates       → JSON 배열 ["2026-06-17", ...]
analysis_YYYY-MM-DD  → JSON { vol:[...], rate:[...], date:"...", indices:{...}, _v: CACHE_VERSION }
```

> ⚠️ **stock_data 스키마를 바꿀 때마다 `src/utils.js`의 `CACHE_VERSION`을 올려야 한다.**
> 캐시에 `_v`가 같이 저장되고, `App.jsx`의 `loadAnalysis()`가 `_v !== CACHE_VERSION`이면
> localStorage를 무시하고 `/api/getData`에서 새로 받아온다. 안 올리면 이미 캐싱된 날짜는
> 새 필드(예: indices)가 영원히 안 보이는 버그가 생긴다 (2026-06-18 실제로 겪은 버그).
> 필드를 추가할 때만이 아니라 **기존 필드의 값 단위를 바꿀 때도 마찬가지** — marketCap/
> tradingVolume을 억원/백만원 단위로 바꾸면서(2026-06-21) `CACHE_VERSION`을 2→3으로 올림.
> 이미 MongoDB에 저장된 과거 문서들은 원 단위로 남아있던 2건(2026-06-18, 19)만 1회성
> 스크립트로 변환해 백필함 — 그보다 더 오래된 문서(06-15~17)는 신기하게도 이미 억원/백만원
> 단위였음(FDR 전환 전 HTS 엑셀 시절 단위가 그대로 남아있던 것으로 추정). 앞으로 다른 과거
> 날짜를 새로 채울 일이 있으면 단위가 섞여있을 수 있으니 값 크기(예: marketCap > 10억)로
> 변환 여부를 먼저 확인할 것.
> **필드 추가/단위 변경 같은 스키마 변화가 아니어도, MongoDB의 기존 문서 내용 자체를 고친
> 경우(예: 스팩 종목 제거 + rank 재정렬, 2026-06-21)에도 똑같이 올려야 한다** — 안 그러면
> 이미 옛 데이터를 캐싱해둔 브라우저는 MongoDB가 고쳐진 뒤에도 옛 localStorage 캐시를 그대로
> 보여준다(실제로 겪음: 스팩 제거 후에도 화면에 계속 보임 → `CACHE_VERSION` 3→4로 해결).
